# OilTech Digest — сводка по проделанной работе

Документ нужен как короткий статус проекта: что уже доведено до рабочего состояния,
что сделано во втором цикле доработок и что пока остается следующим этапом.

## 1. Что уже реализовано

### Источники и ingestion

- загрузка каталога источников из Excel;
- автообнаружение RSS;
- парсинг RSS в PostgreSQL;
- парсинг открытых Telegram-каналов через публичный preview `t.me/s`;
- дедупликация по `url`;
- deterministic prefilter для очевидного нерелевантного шума;
- дозагрузка полного текста статьи по ссылке.

### AI-обработка

- summary статьи;
- AI relevance check;
- тегирование по D01-D18;
- скоринг по настраиваемым критериям;
- учет токенов и стоимости по каждому этапу;
- offline-режим для локальной проверки без OpenAI.

### Продуктовый слой

- Admin UI со списком статей;
- экран источников с ручным добавлением RSS;
- read-only диагностика источников в CLI/API и в экране источников;
- редактирование тегов;
- редактирование критериев скоринга;
- месячный дайджест в виде JSON и HTML email-шаблона;
- Docker Compose с автоциклом ingestion/processing.

## 3. Важные доработки, которые уже добавлены

### Отсев мусора до БД

На этапе `parse` статья может быть отброшена еще до записи в `articles`, если
выглядит как очевидный шум: спорт, бытовые новости, общий офтоп без индустриального сигнала.

Это уменьшает мусор в системе и не тратит AI на заведомо нерелевантные материалы.

### Полный текст статьи

Если RSS отдает только описание, `fetch-full-text` пробует вытащить полный текст
со страницы статьи. Для диагностики в `articles` фиксируются:

- `text_truncated`
- `full_text_status`
- `full_text_error`
- `extraction_method`

### Двухступенчатая релевантность

Сейчас проект использует две ступени:

1. дешевый предфильтр до БД;
2. AI relevance после summary.

Это позволяет одновременно:

- не засорять базу совсем абсурдными материалами;
- не пропускать сложные пограничные кейсы без AI.

### Управление источниками

Экран `Источники` теперь не read-only. Пользователь может:

- смотреть тип источника;
- видеть периодичность и стратегию;
- редактировать `rss_url`;
- включать и выключать источник;
- вручную добавлять новый RSS-источник.

### Дайджест по шаблону

Дайджест можно собрать:

- как JSON через `digest-content`;
- как HTML email через `digest-email` или `--html-output`.

## 4. Экономика AI

В проекте есть:

- `ai_processing_runs` для хранения токенов/стоимости;
- `ai-cost-report` для агрегатов по этапам;
- `ai-article-cost-report` для стоимости полного прогона статьи.

Практический смысл:

- можно оценивать стоимость summary отдельно;
- можно видеть, сколько стоит relevance и сколько он экономит;
- можно быстро считать бюджет на 100, 1000 и более статей.

## 5. Что сейчас является рабочим способом запуска

Основной путь:

```bash
docker compose up -d --build
```

Это поднимает:

- `db`
- `app`
- `scheduler`

Scheduler сам делает базовую инициализацию и затем крутит цикл сбора и обработки.

## 6. Что важно помнить

- проект сейчас заточен в первую очередь под RSS-источники;
- digest пока собирается на лету, а не как полноценная persisted editorial сущность;
- UI уже рабочий, но frontend все еще монолитный и живет в одном `app.html`;
- часть таблиц в схеме создана с запасом под следующие этапы.

## 7. Что логично делать дальше

Наиболее естественные следующие шаги:

1. расширить покрытие источников без RSS;
2. довести monthly digest до отдельной сохраняемой сущности;
3. подключить publish/export слой;
4. при росте интерфейса разнести frontend на более модульную структуру.
