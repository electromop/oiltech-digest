## 6. Задачи (детально: Что / Зачем / Как / Особенности / Сложность)

### P0 — Безопасность SSH `[S]`
**Что:** закрыть открытый `root` + вход по паролю; сервер под активным брутфорсом.
**Зачем:** логи sshd показывают непрерывный перебор паролей (root/admin/test/bot с десятков IP). При слабом пароле — риск компрометации сервера.
**Как:**
1. Поставить fail2ban: `apt-get update && apt-get install -y fail2ban && systemctl enable --now fail2ban`. Jail `sshd` активен по умолчанию (банит после ~5 неудач). Проверить: `fail2ban-client status sshd`.
2. Вход по ключу уже настроен (ключ в `/root/.ssh/authorized_keys`). Проверить с **нормальной сети без VPN**: `ssh root@85.234.107.233` должен пустить без пароля.
3. После подтверждения входа по ключу — отключить пароль:
   `printf 'PasswordAuthentication no\nPermitRootLogin prohibit-password\n' > /etc/ssh/sshd_config.d/90-no-password.conf && systemctl reload ssh`
**Особенности:** НЕ отключать пароль, пока вход по ключу не подтверждён (иначе можно отрезать доступ; веб-консоль остаётся как fallback). Откат: `rm /etc/ssh/sshd_config.d/90-no-password.conf && systemctl reload ssh`.

### P1 — Подтвердить стабильность после OOM-фикса `[S–M]`
**Что:** убедиться, что swap решил проблему и застойные 49 источников снова наполняются.
**Зачем:** «застой» (last_art ≈ 30 мая) совпадает с OOM-периодом. Если после swap они оживают — корень был именно в памяти, и отдельных фиксов по ним не нужно.
**Как:**
1. Форс-цикл: `docker exec oiltech_scheduler python -m oiltech_digest.cli parse --workers 3`, затем `fetch-full-text` и `process`. Через цикл перезапустить вердикт-SQL (§7) — выросло ли число «ОК».
2. Мониторинг памяти: `free -h`, `dmesg | grep -i oom`. Желательно настроить алерт/healthcheck по RAM.
3. Профилактика: при дальнейшем росте нагрузки — задать `mem_limit` контейнерам в `docker-compose.yml` и/или снизить `PARSE_WORKERS`/`AI_PROCESS_LIMIT`.
**Особенности:** swap уже в `/etc/fstab` (переживёт ребут). Главное правило — не собирать образы при работающем стеке.

Извлекать дату/текст/ссылку постов, маппить в `articles` (`source_id`, `title`, `url`, `raw_text`, `published_at`). Использовать существующий `http_client.fetch` (наследует cooldown/прокси).

### P2 — Российские request-источники (0 статей) `[M]`
**Что:** оживить РФ-сайты со стратегией `request`: Росатом, Татнефть, Сбер, СберТех, Минпромторг, МГУ, Узбекнефтегаз, РусЭнерджи, Нефтегазовая вертикаль и др.
**Зачем:** ядро российского нефтегаз-контента сейчас не собирается.
**Как (диагностика per-source обязательна — причины разные):**

**Особенности:** обязательно разделять «гео-блок» и «селекторы» — фиксы разные. Начать с 2–3 приоритетных источников.

### P2 — Международные request-источники с JS/WAF (0 статей) `[L]`

**Зачем:** ключевые мировые источники отрасли.

**Особенности:** `verify=False` SSL-fallback уже есть в `http_client`. Часть `too_short` — реально короткие заметки (не баг).

## 7. Команды-памятка (в веб-консоли сервера, каталог `/root/oiltech-digest`)

```bash
# общая статистика
docker exec oiltech_scheduler python -m oiltech_digest.cli stats

# форс-парсинг (проверить оживление застойных источников)
docker exec oiltech_scheduler python -m oiltech_digest.cli parse --workers 3

# ВЕРДИКТ по источникам: ок / застой / не проходит / выключен
docker exec oiltech_pg psql -U oiltech -d oiltech_digest -c "WITH src AS (SELECT s.id,s.enabled,COUNT(a.id) arts,MAX(a.collected_at) last_art FROM sources s LEFT JOIN articles a ON a.source_id=s.id GROUP BY s.id,s.enabled) SELECT CASE WHEN NOT enabled THEN 'выключен' WHEN arts=0 THEN 'не проходит' WHEN last_art<now()-interval '3 days' THEN 'застой' ELSE 'ОК' END v, COUNT(*) src, COALESCE(SUM(arts),0) art FROM src GROUP BY 1 ORDER BY 2 DESC;"

# ПРОБЛЕМНЫЕ источники детально (что чинить)
docker exec oiltech_pg psql -U oiltech -d oiltech_digest -c "SELECT s.name, s.parse_strategy strat, COUNT(a.id) arts, MAX(a.collected_at)::date last_art FROM sources s LEFT JOIN articles a ON a.source_id=s.id WHERE s.enabled GROUP BY s.id,s.name,s.parse_strategy HAVING COUNT(a.id)=0 OR MAX(a.collected_at)<now()-interval '3 days' ORDER BY arts, s.name;"

# качество full-text
docker exec oiltech_pg psql -U oiltech -d oiltech_digest -c "SELECT COALESCE(full_text_status,'(пусто)') status, COUNT(*) FROM articles GROUP BY 1 ORDER BY 2 DESC;"

# диагностика связности (direct vs proxy)
docker exec oiltech_scheduler python scripts/server_connectivity_check.py

# логи парсера
docker logs -f --tail=100 oiltech_scheduler

# обновление кода на проде (ОБЯЗАТЕЛЬНО с down — иначе OOM при сборке)
docker compose down && git pull origin main && docker compose build && docker compose up -d
```
