## 1. Что это за проект

**OilTech Digest** — внутренняя система для нефтесервисного бизнеса, которая:
1. автоматически собирает статьи из источников (RSS / web-скрейпинг / позже Telegram);
2. прогоняет их через AI-пайплайн: **краткая суть → скоринг релевантности → тегирование**;
3. даёт админ-интерфейс для ручного отбора материалов в **месячный дайджест**;
4. выгружает дайджест в PDF/DOCX и рассылает по корпоративному email-шаблону.

**Заказчик / контекст:** внутренняя рассылка «Блока развития бизнеса»; email-шаблон оформлен в фирменном стиле Газпром нефти (контакт в футере шаблона — `Rodionov.VVL@gazprom-neft.ru`).

**Стадия:** реализован первый кирпич (сбор RSS + БД) и начат backend AI-processing слой
под OpenAI / ChatGPT API: суть статьи, тегирование, скоринг, учёт токенов и JSON-черновик
дайджеста.

## 3. Архитектура (источник истины — `docs/architecture.md`)

Пайплайн: `Scheduler → Parser (RSS / web / позже Telegram) → нормализация → дедупликация → AI-суть → AI-скоринг → AI-тегирование → карточка статьи → Admin UI → месячный дайджест → экспорт PDF/DOCX`.

**Схема БД — 11 нормализованных таблиц** (полный DDL — в `docs/architecture.md` §9 и реализован в `oiltech_digest/db/schema.sql`):
`sources`, `articles`, `article_cards`, `scoring_criteria`, `article_scores`, `article_score_items`, `tags`, `article_tags`, `monthly_digests`, `monthly_digest_items`, `export_jobs`.

**Интерфейс (макет — `web/product_mockups.html`):** 4 экрана — «Все статьи», «Месячный дайджест», «Скоринг», «Теги».

## 5. Доменные правила (соблюдать во всех задачах)

- **Telegram — только ранний сигнал**, не первоисточник. В финальный дайджест новость попадает лишь при подтверждённом источнике (URL + дата).
- **Дата публикации** новости должна входить в период выпуска дайджеста.
- **Email:** HTML-шаблон письма НЕ редактировать — менять только `digest_content.json`.
- **Скоринг:** сумма весов критериев = ровно 100%; русские ключевые слова нормализуются в английские для зарубежных источников.
- **Теги** иерархические (родитель/подтег), статья группируется по верхнеуровневому тегу.

## 7. Что выполнено: Issue #1 — сбор RSS + БД

**Коммит `2678c4b` в ветке `dev`** (21 файл). Реализовано с нуля (репозиторий был без кода).

### Структура кода
```
oiltech_digest/
  config.py                 # REPO_ROOT, DATABASE_URL, SOURCES_XLSX, константы парсера
  db/
    schema.sql              # DDL всех 11 таблиц (Postgres) + индексы
    connection.py           # psycopg connect, init_db()
    repository.py           # CRUD sources + articles (ON CONFLICT)
  ingestion/
    excel_seed.py           # Excel (Sources_Expanded) → sources
    rss_discovery.py        # автообнаружение RSS по сайту источника
    http_client.py          # fetch() с retry/backoff/SSL-fallback; probe() без ретраев
    rss_parser.py           # ленты → нормализованные записи → articles
    normalize.py            # clean_html, parse_date, extract_image, compute_content_hash
  cli.py                    # init-db / seed-sources / discover-rss / parse / stats
tests/                      # офлайн-тесты (normalize, discovery) — 9 шт.
docker-compose.yml          # PostgreSQL 16
requirements.txt / requirements-dev.txt / .env.example / README.md
```

### Результаты прогона на реальных данных
- **120 источников** засеяно из Excel.
- **Автообнаружение RSS:** 22 рабочих ленты (`rss`), 83 без RSS (`request` — кандидаты на web-парсер), 15 `telegram` (пропущены).
- **807 статей** собрано, **0 ошибок** парсинга.
- **Дедупликация работает:** повторный `parse` → 0 новых. URL-дубли = 0, `content_hash` заполнен у всех.
- **Тесты:** 9/9 зелёные.

### Ключевые технические решения
- **Схема БД** — нормализованная по `architecture.md`; создаются сразу все 11 таблиц, на этом этапе наполняются `sources` и `articles`. Типы адаптированы под Postgres (`IDENTITY`, `TIMESTAMPTZ`, `BOOLEAN`, `NUMERIC`, `JSONB`).
- **Ключ уникальности источника — `(name, source_type)`**, не просто `name`: один бренд может иметь сайт и Telegram-канал с одинаковым именем (Neftegaz.ru, РБК Энергетика, Интерфакс ТЭК) — это разные источники.
- **Дедуп статей:** жёстко по `url` (UNIQUE + `ON CONFLICT DO NOTHING`) + мягкий `content_hash = sha256(нормализованные title|host+path без query/utm)` (индексируется, пока НЕ UNIQUE — жёсткое схлопывание перепечаток оставлено на этап нормализации pipeline).
- **Автообнаружение RSS** трёхступенчатое: `<link rel="alternate">` на главной → перебор типичных путей (`/feed`, `/rss`, `/rss.xml`, …) → валидация feedparser (лента засчитывается только при наличии entries).
- **`probe()` без ретраев** для зондирования RSS-кандидатов (404 на несуществующих путях ожидаемы — ретраи с backoff там лишь тормозили бы).
- **SSL-fallback `verify=False`** — только запасной путь при `SSLError`, с предупреждением в лог (нужен проблемным сайтам; окружение в целом исправно — `certifi` работает).
- **Тематический фильтр НЕ применяется** (релевантность — задача скоринга #7), **окно свежести опционально** (`--max-age-days`, по умолчанию выкл — продукт месячный, копит статьи).

### Как запустить (runbook)
```bash
cd oiltech-digest
cp .env.example .env          # вписать пароль POSTGRES_PASSWORD (локальная среда)
colima start                  # или запустить Docker Desktop
docker-compose up -d db       # PostgreSQL 16
python3 -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt
python -m oiltech_digest.cli init-db
python -m oiltech_digest.cli seed-sources
python -m oiltech_digest.cli discover-rss     # ~10 мин (сетевой обход, таймауты гос-сайтов)
python -m oiltech_digest.cli parse
python -m oiltech_digest.cli stats
```
Реальный `.env` (с паролем) НЕ в гите; БД живёт в Docker-томе `pgdata`. На чистой машине нужны `docker-compose up -d db` + `init-db`.

## 9. Известные нюансы и наблюдения

- **`discover-rss` идёт ~10 мин**: источники без RSS перебирают все ~16 путей, гос-сайты дают connect-timeout 10 с. Кандидат на оптимизацию (ранний отказ по домену, кэш).
- **Шумные / архивные ленты**: ДРТ отдал 320 записей, McKinsey — кроссворды и нерелевант, МФТИ — культурные новости. Это ожидаемо (фильтр релевантности не ставили) — отсечёт скоринг #7.
- **Эвристика языка** в парсере грубая (по группе источника РФ/СНГ→ru, международные→en). Точное определение языка — на будущее (важно для #4/#10).
- **`content_hash`** пока не UNIQUE: кросс-источниковые перепечатки только помечаются, не схлопываются (решение — на этапе нормализации pipeline).
- **Рабочее соглашение по процессу:** при блокирующей проблеме (особенно инфраструктура) — не «долбиться», а спрашивать владельца. Переиспользуемый код из прототипа — проверять и валидировать на реальных данных, не копировать слепо.
